## Game

### PixelGuardian
制作: Unity / 制作期間: 半年 / 制作人数: 5人

タッチで敵を倒す 2D アクションゲームです。iPad 向けに制作しました。

**担当:** クライアントエンジニア（負荷対策・データ管理設計）

**課題:**
iPad という端末性能の制約がある中で、多数の敵が迫るパニック感を維持する必要がありました。また、プランナーが Unity を直接操作できず、パラメータ調整のたびにエンジニアへの依頼が発生していました。

**判断:**
Profiler で計測したところ、敵オブジェクトの生成・破棄コストが負荷の主因と判断しました。同時に、パラメータ調整の手間を減らすため、リソースとデータをコードから切り離す構成を採ることにしました。

**実装:**
- ObjectPooling を導入し、敵の生成・破棄コストを削減
- Addressables と JSON / CSV による外部データ管理を実装し、コード変更なしでリソースやパラメータを調整できる構成にした

**成果・学び:**
フレームレートの安定化と、プランナーによる自立した調整フローを同時に実現しました。体験品質と制作フローの両方を改善対象として考える視点が身につきました。

### Falling Slime
制作: Unreal Engine 5

東京ゲームショウ 2024 の学内ブースで展示した作品です。

**担当:** クライアントエンジニア（ギミック実装・負荷軽減）

**課題:**
展示機のスペック内で安定稼働させることが必要でした。フレームレートが落ちると展示中のプレイ体験が損なわれるため、パフォーマンスの確保は体験品質に直結していました。

**判断:**
フィールドを分割し、非同期ロードで必要な部分だけ読み込む方針を選びました。また、負荷の原因になりうる実装についてプランナー・3D モデラーに事前共有し、設計段階での対策を促しました。

**実装:**
- フィールドの非同期ロードを実装し、不要なフィールドを読み込まない構成にした
- 一定時間で切り替わる床、コレクションアイテムのギミックを実装した

**成果・学び:**
展示環境では安定動作そのものが体験品質の一部であると実感しました。エンジニアから非エンジニアへの事前共有が、後工程の手戻りを減らすうえで有効だと学びました。

[動画はこちらから確認できます（YouTube へ遷移します）](https://www.youtube.com/watch?v=WjKkpfJfV7s)

### AI彼女
制作: Python / Docker

AI で彼女を作る試みです。ChatGPT のプロンプト操作、GPTs と試みを重ねましたが、どちらもプロンプトベースの操作のため会話の文脈情報が欠損しやすく、また音声で応答することができませんでした。

現在は Python と TTS（音声合成）を組み合わせた構成に切り替え、「声で会話できる彼女」を目指して制作中です。モデルの選定・プロンプト設計・音声生成の統合を自分で行い、AI を「使いこなす」実装を意識しています。

### 開発支援ツール
制作: Unity / C#

ゲーム開発の中で、改善点の共有・デバッグ・データ編集に手間がかかるという課題から着手しました。

**担当:** 個人開発（設計・実装全般）

**課題:**
開発中のゲームで、改善点の共有・デバッグ・データ修正のたびにエンジニアへの依頼や手作業が発生していました。

**判断:**
ゲーム本体の実装だけでなく、制作を支える環境を整えることにも価値があると考えました。生成 AI を実装補助として使い、課題設定や機能要件の整理に重点を置きました。

**実装:**
- Unity Editor から Missing 参照を自動検出するツールを試作（[SM_MissingScanner](https://github.com/UchinoKoki/SM_MissingScanner)）
- JSON を GUI で編集できるデータ管理ツールを試作（[JsonTool](https://github.com/UchinoKoki/JsonTool)）

**成果・学び:**
ゲーム本体だけでなく、制作を支える環境にも関心があることに気づきました。ツール開発を通じて、「誰が・何を・どこで困っているか」を先に設定することの重要さを学んでいます。
